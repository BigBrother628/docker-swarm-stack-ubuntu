# Docker Swarm Stack: Ubuntu Environment
Репозиторий содержит конфигурацию для развертывания отказоустойчивого стека в Docker Swarm, включающего сервис на базе образа **ubuntu:22.04**, а также базы данных **PostgreSQL** и **Redis**.

### Что было сделано:

* [cite_start]**Проектирование архитектуры стека**: Разработан декларативный файл конфигурации `stack.yml`, описывающий три взаимосвязанных сервиса с учетом зависимостей и сетевой изоляции[cite: 3, 4, 15].
* [cite_start]**Реализация сетевой безопасности**: Настроены две независимые overlay-сети (`db-postgres-net` и `ds-redis-net`), что позволило изолировать базы данных друг от друга, оставив доступ к ним только для прикладного сервиса[cite: 5, 6].
* [cite_start]**Оптимизация доставки обновлений**: Внедрена стратегия **Rolling Update** (параметр `start-first`), гарантирующая доступность сервиса без простоев при выпуске новых версий кода[cite: 11].
* **Управление ресурсами и надежностью**: 
    * [cite_start]Установлены жесткие лимиты на потребление ресурсов (1 CPU, 500MB RAM), что предотвращает деградацию всей системы при сбоях в отдельном контейнере[cite: 13].
    * [cite_start]Настроена ротация системных логов (5MB, 1 файл) для предотвращения переполнения дискового пространства[cite: 14].
* [cite_start]**Настройка персистентности данных**: Реализован механизм проброса (volume) логов приложения на хост-систему для обеспечения возможности внешнего аудита и сохранения данных после удаления контейнеров[cite: 12].
* [cite_start]**Обеспечение высокой доступности (HA)**: Сконфигурирована работа минимум 2-х реплик сервиса с распределением по нодам кластера, имеющим метку `SERVERTYPE=worker`[cite: 9, 15].
* [cite_start]**Автоматизация контекста**: Использованы шаблоны Swarm для динамической передачи имени физической ноды (`HOSTNAME`) внутрь каждого экземпляра контейнера[cite: 16].

### Достигнутые результаты:

1.  **Zero Downtime**: Система способна обновляться и переносить сбои отдельных узлов без прерывания обслуживания пользователей.
2.  **Эффективное использование ресурсов**: За счет лимитов и меток размещения (placement constraints) достигнута стабильная работа на целевом оборудовании.
3.  **Безопасность "по умолчанию"**: Изоляция на уровне сетей минимизирует риски при компрометации одного из узлов.
4.  [cite_start]**Удобство отладки**: Прозрачный доступ к логам на хосте и настроенная среда внутри контейнеров позволяют инженерам оперативно проводить диагностику[cite: 7].
