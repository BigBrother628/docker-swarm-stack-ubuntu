# Docker Swarm Stack: Ubuntu Environment
Репозиторий содержит конфигурацию для развертывания отказоустойчивого стека в Docker Swarm, включающего сервис на базе образа **ubuntu:22.04**, а также базы данных **PostgreSQL** и **Redis**.

### Что было сделано:

* **Проектирование архитектуры стека**: Разработан декларативный файл конфигурации `stack.yml`, описывающий три взаимосвязанных сервиса с учетом зависимостей и сетевой изоляции[cite: 3, 4, 15].
* **Реализация сетевой безопасности**: Настроены две независимые overlay-сети (`db-postgres-net` и `ds-redis-net`), что позволило изолировать базы данных друг от друга, оставив доступ к ним только для прикладного сервиса[cite: 5, 6].
* **Оптимизация доставки обновлений**: Внедрена стратегия **Rolling Update** (параметр `start-first`), гарантирующая доступность сервиса без простоев при выпуске новых версий кода[cite: 11].
* **Управление ресурсами и надежностью**: 
* Установлены жесткие лимиты на потребление ресурсов (1 CPU, 500MB RAM), что предотвращает деградацию всей системы при сбоях в отдельном контейнере[cite: 13].
* Настроена ротация системных логов (5MB, 1 файл) для предотвращения переполнения дискового пространства[cite: 14].
* **Настройка персистентности данных**: Реализован механизм проброса (volume) логов приложения на хост-систему для обеспечения возможности внешнего аудита и сохранения данных после удаления контейнеров[cite: 12].
* **Обеспечение высокой доступности (HA)**: Сконфигурирована работа минимум 2-х реплик сервиса с распределением по нодам кластера, имеющим метку `SERVERTYPE=worker`[cite: 9, 15].
* **Автоматизация контекста**: Использованы шаблоны Swarm для динамической передачи имени физической ноды (`HOSTNAME`) внутрь каждого экземпляра контейнера[cite: 16].

### Достигнутые результаты:

1.  **Zero Downtime**: Система способна обновляться и переносить сбои отдельных узлов без прерывания обслуживания пользователей.
2.  **Эффективное использование ресурсов**: За счет лимитов и меток размещения (placement constraints) достигнута стабильная работа на целевом оборудовании.
3.  **Безопасность "по умолчанию"**: Изоляция на уровне сетей минимизирует риски при компрометации одного из узлов.
4.  **Удобство отладки**: Прозрачный доступ к логам на хосте и настроенная среда внутри контейнеров позволяют инженерам оперативно проводить диагностику[cite: 7].

### Гайд по проверке
1. **Подготовьте окружение**
`docker swarm init` и `docker node update --label-add SERVERTYPE=worker $(docker node ls -q)`
2. **Склонируйте репозиторий**
   `git clone https://github.com/BigBrother628/docker-swarm-stack-ubuntu.git`
3. **Разверните стек:** `docker stack deploy -c stack.yml my_stack`
4. **Проверьте статус сервисов:** `docker stack services my_stack`
5. **Проверьте переменные окружения и подключения к БД:**
   CONTAINER_ID=$(docker ps --filter "name=my_stack_app" -q | head -n 1)
     * **Заходим внутрь контейнера**: `docker exec -it $CONTAINER_ID bash`
6. **Внутри контейнера выполните установку утилит и тесты:**
     * `apt-get update && apt-get install -y postgresql-client redis-tools`
     * `pg_isready -h db -p 5432`
     * `redis-cli -h redis ping`
     * `exit`
7. **Проверьте, что лимиты выставлены корректно:**
   * `docker inspect $CONTAINER_ID --format '{{json .HostConfig.Resources}}'`
     Должны увидеть:
      * NanoCpus: `1000000000`
      * Memory: `524288000`
8. **Проверьте логи (volumes):**
   * `docker exec $CONTAINER_ID sh -c "echo 'Healthcheck passed' > /var/log/app/health.log"`
9. **Проверьте ротацию логов**
   * `docker inspect --format='{{json .HostConfig.LogConfig}}' $CONTAINER_ID`
10. **Очистите ресурсы:** `docker stack rm my_stack`
